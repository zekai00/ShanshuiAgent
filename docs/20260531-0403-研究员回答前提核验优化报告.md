# 研究员回答前提核验优化报告

生成时间：2026-05-31 04:03 CST

## 优化目标

错误前提专项评测显示，检索层可以改善证据召回，但不能保证回答一定“不顺着错误前提编造”。因此本轮优化 researcher 的回答规则，让回答层显式执行前提核验。

## 改动内容

修改文件：`src/agent/prompts/researcher_prompt.yaml`

新增“前提核验规则”：

1. 对“是否/是不是/能否”等问题先判断历史前提是否成立。
2. 遇到时代错置、现代技术错置、人物/流派混淆、作品归属混淆，开头必须直接说明“不成立”或“这个前提不成立”。
3. 错误前提题至少组织两类证据：正确对象/概念的证据，以及被混淆对象真实时代、流派或性质的证据。
4. 固定输出结构为“前提判断、依据与解释、可用于创作/研究的要点、来源”。

## 影响范围

这次没有修改 LangGraph 编排，也没有修改工具调用方式。原因是当前最大问题不是节点流转，而是 researcher 最终回答格式缺少强约束；用提示词先加结构约束，风险更小，也更容易和后续 SFT 数据格式对齐。

## 后续评测建议

下一轮应做“回答质量评测”，而不只是检索命中评测。建议新增 3 个维度：

- 前提纠错：错误前提题是否第一句明确纠错。
- 证据引用：是否包含文献名、页码或证据块ID。
- 不编造：资料不足时是否承认证据不足，而不是补造结论。

这部分适合之后用 SFT 数据固化；暂时不需要 PPO/GRPO。
