# ChineseLandscape 训练与对齐下一步计划

## 先回答：DPO 是不是 RL？

结论：DPO 属于 RLHF/偏好对齐谱系，但它不是 PPO 那种在线强化学习训练。

更准确地说：

- 从理论来源看，DPO 可以由带 KL 约束的 RLHF 目标推导出来，所以很多人会把它归在 RLHF 或 preference optimization 里。
- 从工程训练方式看，DPO 不需要在线 rollout，不需要单独训练 reward model，也不需要 PPO 式 policy gradient 更新。
- 它使用离线偏好样本 `(prompt, chosen, rejected)` 直接优化模型，让模型提高 chosen 相对 rejected 的概率。

所以如果“RL”指广义的人类偏好强化学习路线，DPO 算；如果“RL”指 PPO/GRPO 这类在线采样、奖励打分、策略优化流程，DPO 不算同一种工程范式。

对本项目而言，DPO 应该被理解为“离线偏好对齐工具”，不是优先级最高的在线 RL 方案。

## 当前判断

ChineseLandscape 现在最需要的不是立刻上 RL，而是先把 RAG 回答链路、训练数据格式和评测闭环打牢。

当前 `scripts/experiments/generate_dpo_data.py` 的方向是对的：它想让模型偏好“基于文献、少幻觉、有出处意识”的回答。但现有实现有一个关键问题：

```json
{
  "instruction": "用户问题",
  "input": "",
  "chosen": "基于检索文献生成的好回答",
  "rejected": "坏回答"
}
```

`chosen` 是基于检索文献生成的，但训练样本的 `input` 为空。模型训练时看不到文献上下文，却被要求输出像看过文献一样的答案，这会把模型推向“有文献腔的幻觉”，而不是 grounded answering。

必须先把检索上下文放进 `input` 或 prompt 中。

## 总路线

推荐路线：

```text
工程清理与评测集
  -> RAG/SFT 数据构造
  -> SFT 训练 researcher/answer 模型
  -> DPO 精修回答偏好
  -> 仅对可自动验证任务尝试 GRPO
  -> 暂不使用 PPO
```

核心原则：

1. 能用规则修的，先用规则修。
2. 能用 prompt 和 RAG 修的，先用 prompt 和 RAG 修。
3. 能用 SFT 学会的，不急着上 DPO。
4. 只有存在可自动打分的行为目标时，才考虑 GRPO/RL。
5. PPO 放到最后，除非已经有稳定 reward model 和明确的 agent policy 训练目标。

## 阶段 0：先修训练前提

目标：确保后续训练真的能作用于在线系统，而不是离线做了一套无关实验。

任务：

1. 确认当前在线系统实际使用哪个回答模型。
   - `src/agent/main_nodes.py` 当前使用 DeepSeek API 做 supervisor/researcher/artist/chatter。
   - 如果训练 Qwen LoRA，需要明确把训练后的模型接入 researcher 或最终回答节点。

2. 清理敏感配置。
   - 把脚本中的 API key、HF token、LangSmith key 全部迁移到 `.env`。
   - 已经暴露过的 token 应轮换。

3. 统一端口与配置。
   - 离线入库阶段：`8000/v1` 是本地文本 LLM。
   - 在线检索阶段：`8000/retrieve` 是 FastAPI 检索服务。
   - 建议用配置文件区分 `LOCAL_LLM_PORT`、`RETRIEVAL_PORT`，避免冲突。

4. 建立最小可重复运行脚本。
   - `make ingest`
   - `make graph`
   - `make retrieval`
   - `make chat`

交付物：

- `.env.example`
- `docs/RUNBOOK.md`
- 统一配置项
- 明确“微调模型接入点”

验收标准：

- 能从同一套配置启动检索服务和终端智能体。
- 能确认训练后的模型在哪个节点被调用。

## 阶段 1：建立评测集

目标：先知道系统哪里差，再决定 SFT、DPO 或 RL。

建议建立 `data/eval/landscape_eval.jsonl`。

每条样本包含：

```json
{
  "id": "eval_0001",
  "question": "黄公望的《富春山居图》主要使用了什么皴法？",
  "task_type": "factual_qa",
  "expected_entities": ["黄公望", "富春山居图", "披麻皴"],
  "gold_sources": ["xxx.pdf"],
  "must_cite": true,
  "notes": "答案必须基于检索文献，不允许凭空扩展。"
}
```

覆盖类型：

- `factual_qa`：事实问答，如画家、作品、技法。
- `comparison`：比较问答，如北宋/南宋山水画差异。
- `concept_explain`：术语解释，如三远法、留白、披麻皴。
- `timeline`：朝代流变。
- `evidence_missing`：资料库没有依据时的拒答。
- `artist_prompt`：把考据结果转为生图 prompt。
- `routing`：supervisor 应分发给 researcher、artist 或 chatter。

建议规模：

- 第一版：100 条。
- 第二版：300-500 条。
- 稳定后：1000 条以上。

指标：

- 检索 `Recall@3/5/10`
- 引用准确率
- 回答事实一致性
- 幻觉实体数量
- 无证据拒答率
- supervisor 路由准确率
- artist prompt 是否包含关键技法/构图/题材

交付物：

- `data/eval/landscape_eval.jsonl`
- `scripts/eval/evaluate_retrieval.py`
- `scripts/eval/evaluate_answers.py`
- `config/eval/rubric.yaml`

验收标准：

- 每次改 prompt、检索器或模型后，都能跑同一套评测。
- 能明确看到失败来自检索、路由、回答还是生图 prompt。

## 阶段 2：先做 SFT

目标：让模型学会“给定检索文献后，如何专业回答”。

SFT 应优先训练 researcher 或最终回答模型，而不是训练整个 agent。

推荐样本格式：

```json
{
  "instruction": "请基于给定文献回答用户问题，并在关键结论后标注出处。",
  "input": "【检索文献1】...\n【检索文献2】...\n【用户问题】黄公望的《富春山居图》主要使用了什么皴法？",
  "output": "《富春山居图》通常被认为以披麻皴为主要山石表现方法..."
}
```

SFT 数据来源：

1. 从现有 Milvus 检索结果构造问题。
2. 用强模型基于检索文献生成标准答案。
3. 人工抽查并修正高价值样本。
4. 从 `data/fintune_results/` 中筛选可用问答，但必须补齐文献上下文。

建议先做四类 SFT 数据：

- 文献 grounded QA。
- 术语解释。
- 比较型回答。
- 考据转绘画 prompt。

不要一开始就训练：

- 多轮 agent 策略。
- 工具调用策略。
- 主观审美判断。

建议规模：

- 快速验证：500-1000 条。
- 第一版可用：3000-5000 条。
- 更稳定版本：10000 条以上。

交付物：

- `data/alignment/sft_researcher.jsonl`
- `data/alignment/sft_artist_prompt.jsonl`
- `scripts/datasets/build_sft_data.py`
- LLaMA-Factory 数据集注册配置

验收标准：

- 在同样检索上下文下，SFT 模型比 base 模型更少幻觉。
- 引用格式更稳定。
- 回答更像中国美术史研究员，而不是泛泛百科回答。

## 阶段 3：再做 DPO

目标：在 SFT 模型已经会回答的基础上，进一步让它偏好更严谨的回答。

DPO 适合处理“两个回答都像答案，但一个更好”的情况。

推荐 DPO 样本格式：

```json
{
  "instruction": "请基于给定文献回答用户问题，并标注出处。",
  "input": "【检索文献】...\n【用户问题】...",
  "chosen": "有出处、少幻觉、准确解释术语的回答。",
  "rejected": "没有出处、泛泛而谈、或引入资料外事实的回答。"
}
```

`rejected` 的类型要有针对性：

- 引用不存在的 PDF。
- 把朝代说错。
- 把画家和作品关系说错。
- 技法错配，例如把披麻皴说成斧劈皴。
- 没有使用检索资料。
- 过度扩展资料库里没有的内容。
- 回答风格网文化、敷衍化。
- 用户问无证据问题时仍然硬答。

对当前脚本的修改重点：

1. `input` 必须包含检索上下文。
2. `chosen` 和 `rejected` 最好共享同一个 prompt/context。
3. `rejected` 不要只做“故意很烂”的回答，也要做“看起来合理但有细节错误”的 hard negative。
4. 记录 `source_file`，便于后续检查引用是否真实。

交付物：

- `scripts/datasets/build_dpo_data.py`
- `data/alignment/dpo_researcher.jsonl`
- DPO 前后对比报告

验收标准：

- DPO 模型在评测集中更少出现无证据断言。
- 引用准确率提升。
- 不降低基础问答能力。
- 不牺牲过多回答完整性。

## 阶段 4：什么时候需要 GRPO/RL

只有当目标可以自动打分时，才值得上 GRPO。

本项目适合 GRPO 的任务：

### 1. 引用与证据一致性

奖励项：

- 回答中的文献名必须来自检索结果。
- 回答中的画家、朝代、作品、技法必须能在上下文或图谱中找到支持。
- 没有证据时应拒答或说明资料库未命中。

可能的奖励函数：

```text
reward =
  0.30 * citation_validity
+ 0.25 * context_entity_support
+ 0.20 * answer_format_score
+ 0.15 * no_unsupported_claims
+ 0.10 * helpfulness
```

### 2. 查询改写和多步检索

奖励项：

- 改写后的 query 提高命中文献的 rerank score。
- 多步检索能找回 gold source。
- 不进行无意义重复检索。

适用场景：

- 用户问题很口语化。
- 用户问题缺少朝代、画家、术语。
- 第一次检索结果低分，需要自动重查。

### 3. 工具调用策略

奖励项：

- 专业问题必须调用 `search_landscape_literature`。
- 闲聊不调用检索。
- 画图需求先考据再生图。
- ComfyUI 不在线时不伪造图片结果。

但这一项可以先用路由 SFT 和规则解决，不急着 RL。

不建议用 GRPO/RL 的任务：

- 直接优化“画得好不好”。
- 直接优化主观审美。
- 替代检索器训练。
- 修复 PDF 解析、实体抽取、端口冲突、工程配置问题。

## 为什么暂不建议 PPO

PPO 当前不适合本项目的主要原因：

- 需要可靠 reward model。
- 在线 rollout 成本高。
- 训练稳定性和超参成本高。
- 你的主要问题目前是 grounded RAG、数据格式和系统工程问题，不是必须靠 PPO 才能解决的 policy learning 问题。

除非后续满足以下条件，否则不建议上 PPO：

- 已经有稳定 SFT 模型。
- 已经有高质量 reward model。
- 已经有可重复的多轮 agent 环境。
- 已经证明 DPO/GRPO 不能满足需求。

### 第 3 周：SFT 训练与接入

任务：

- 用 LLaMA-Factory 训练 researcher LoRA。
- 在固定评测集上比较 base、SFT、当前 DeepSeek 链路。
- 将 SFT 模型接入 researcher 节点进行本地测试。

产出：

- SFT LoRA。
- SFT 评测报告。

### 第 4 周：DPO 精修

任务：

- 基于 SFT 模型生成候选回答。

- 写 `scripts/datasets/build_dpo_data.py`。
- 训练 DPO LoRA。
- 对比 SFT vs SFT+DPO。

产出：

- `data/alignment/dpo_researcher.jsonl`
- DPO LoRA。
- DPO 评测报告。

### 第 5 周以后：GRPO 小实验

任务：

- 只选择一个可验证任务，例如“引用必须来自检索结果”。
- 写 reward function。
- 小规模训练，不动主系统。
- 如果收益明确，再扩展到 query rewrite 或工具调用策略。

产出：

- GRPO 可行性报告。
- 是否继续投入 RL 的决策。

## 优先级清单

P0：

- 修复训练样本不包含检索上下文的问题。
- 建立评测集。
- 明确训练模型接入点。

P1：

- 做 researcher SFT。
- 做 artist prompt SFT。
- 修正 DPO 数据构造脚本。

P2：

- 做 DPO 精修。
- 建立 hard negative 生成策略。
- 加入引用与实体一致性自动评测。

P3：

- 尝试 GRPO。
- 研究多步检索和工具调用策略训练。

暂缓：

- PPO。
- 端到端训练整个多智能体系统。
- 主观审美 RL。

## 最终建议

当前系统没有“本质用错 DPO”，但 DPO 现在用得偏早，而且数据格式需要修正。

最合理的下一步是：

1. 先建立评测集。
2. 再做基于检索上下文的 SFT。
3. 用 DPO 修正 SFT 后的回答偏好。
4. 只有对引用、证据一致性、查询改写、工具调用这类可验证目标，才考虑 GRPO。
5. 暂不使用 PPO。
