# PDF 构建 RAG 系统链路诊断与重构报告（2026 新版）

生成时间：2026-05-30 13:36 CST

## 结论

依据 2025-2026 年最新资料重新判断后，我的结论比上一版更明确：

**你当前 RAG 系统的检索核心不是错的，但 PDF 证据层结构不够合理。现在最该做的是重构“文档解析、证据存储、引用追踪、指定文献约束”，而不是继续训练模型。**

2026 年相关资料并没有说明“RAG 已经过时”。相反，趋势是：

- 从简单向量检索，走向 hybrid retrieval + reranker。
- 从孤立 chunk，走向 layout-aware、cross-page、document graph。
- 从固定 top-k，走向 query-aware / agentic / adaptive retrieval。
- 从“回答像对”，走向 source document、chunk、page、citation 可验证。
- 从把图谱当知识源，走向用图谱指导检索，再回到底层证据。

对你的项目来说，这意味着：

1. 保留 Milvus、BGE-M3、dense/sparse/ColBERT、RRF、reranker。
2. 暂停 DPO/PPO/GRPO 和大规模 SFT。
3. 先重构 PDF canonical document store。
4. 增加 source-aware retrieval。
5. 拆开 raw evidence 与 LLM 生成的 contextual enrichment。
6. 增加 citation verifier。
7. 重构后再用 reviewed 评测集评估。

## 2026 最新资料对本项目的新增判断

### 1. Hybrid retrieval + reranker 仍然是正确方向

2026 年 `From BM25 to Corrective RAG` 对 text-and-table 文档检索做了系统 benchmark。它的结论对你的系统非常重要：

- 单一路径检索不够。
- hybrid retrieval + neural reranker 的两阶段方案明显优于单阶段方案。
- BM25 在精确术语、数字、表格类文档里可能超过 dense retrieval。
- query expansion、HyDE、multi-query 对精确查询的收益有限。
- contextual retrieval 有稳定收益。

对应到你的项目：

当前使用 BGE-M3 dense/sparse、ColBERT、RRF、reranker 是合理的。  
但只依赖 BGE sparse 不够。中国山水画语料里有大量精确名词：

- 画家：倪瓒、黄公望、董其昌、石涛、王时敏。
- 术语：披麻皴、斧劈皴、计白当黑、三远法。
- 文献题名：《明代山水画中桥梁意象研究》。
- 画作名：《富春山居图》《溪山行旅图》。

所以你应该补 Milvus native BM25/full-text search 或独立 BM25 索引，让题名、术语、画家名可以稳定命中。

### 2. 2026 的 PDF/文档 RAG 更强调 layout-aware 和 cross-page

LAD-RAG 的 2026 修订版明确指出，视觉丰富文档问答不能只把内容切成孤立 chunk。文档的版面结构、跨页依赖、页面关系会影响检索和问答。

这和你的项目高度相关，因为你的 PDF 不是简单纯文本：

- 有论文版式。
- 有页眉页脚。
- 有脚注、图注、插图。
- 有被 VLM 裁剪出的古画图片。
- 有可能跨页讨论同一画家、同一画作、同一概念。

当前系统把 PDF 逐页转图给 VLM，再得到文字和插图描述。这个方向能处理扫描件，但缺少结构化 page/element/chunk 层：

- 没有 page_number。
- 没有 bbox。
- 没有 section_title。
- 没有 paragraph_id。
- 没有 image_id 与 chunk 的关系。
- 没有跨页关系。

2026 的判断是：**孤立 chunk 是不够的，尤其不适合 PDF 和图文混排材料。**

因此，上一版提出的 canonical document store 不是“锦上添花”，而是 P0 必做。

### 3. Agentic RAG 不是“让模型更自由”，而是让检索流程可控

2026 版 Agentic RAG survey 强调 agentic RAG 的核心不是炫技，而是：

- query reformulation
- planning
- tool use
- multi-agent coordination
- adaptive retrieval
- evaluation、memory、governance

对你的系统来说，不应马上做复杂多智能体。你真正需要的是三个受控的小 agent / tool step：

1. Query Parser  
   解析用户是否指定文献、画家、朝代、术语、作品名、错误前提。

2. Retrieval Planner  
   根据问题类型决定：
   - 是否强制 source filter
   - 是否启用 BM25
   - 是否启用 graph expansion
   - 是否需要同一文献多页证据
   - 是否需要图像 evidence

3. Citation Verifier  
   回答生成后检查：
   - 引用是否来自本轮 evidence
   - PDF 是否存在
   - 页码/chunk_id 是否存在
   - 指定文献题是否引用了指定文献
   - 错误前提题是否先纠错

这比直接训练一个小模型“学会研究员回答”更靠谱。

### 4. GraphRAG 的作用应被收窄

2026 Graph-RAG survey 的重点是可信推理、结构化检索、图谱引导，而不是让图谱替代原文证据。

你的系统现在 Neo4j 作为第四路召回是可以的，但应该明确：

- 图谱用于实体扩展、路径发现、跨文献关系召回。
- 最终回答仍必须引用 PDF raw chunk。
- 图谱节点本身不应作为最终事实依据，除非它也能追溯到 PDF chunk。

所以当前 Neo4j 离线时系统能降级，这不是最大问题。最大问题还是底层 PDF evidence 没有 page/chunk provenance。

### 5. 官方工具链也在强化 metadata/filter/citation

Milvus 官方文档已经支持 full-text search/BM25，并说明可以和 dense vector search 结合。  
LangChain 文档强调 RAG 应保存 retrieved documents 和 metadata。  
OpenAI Retrieval / Vector Stores 文档也把 vector store file、attributes/filter、chunked indexed files 作为核心对象。

这些都指向同一个工程原则：

**RAG 系统不应该只传字符串上下文，而应该传结构化 evidence object。**

你的当前 `search_landscape_literature` 返回的是字符串：

```text
来源文献: xxx.pdf
内容详情: ...
```

这能跑通 demo，但不适合严肃的学术 RAG。下一版应该返回结构化 evidence pack，再由 Researcher 生成回答。

## 当前系统与 2026 最佳实践的差距

### 差距 1：没有 canonical document store

当前 Milvus 基本承担了“索引”和“证据存储”两个角色。  
但向量库不应该是唯一真相源。

应该新增：

```text
data/processed/documents/
  documents.jsonl
  pages.jsonl
  elements.jsonl
  chunks.jsonl
  images.jsonl
  manifest.json
```

Milvus 只负责检索索引。真正的 evidence 应从 canonical store 取。

### 差距 2：没有 page/chunk 级引用

当前引用粒度：

```text
[文献: 《xxx.pdf》]
```

新版应至少做到：

```text
[文献: 《xxx.pdf》, 页: 12, chunk: doc_abc_p12_c03]
```

未来如果要支持图像，还应支持：

```text
[图像: image_abc_p12_02, 文献: 《xxx.pdf》, 页: 12]
```

### 差距 3：指定文献没有检索约束

用户问“根据《某文献》”，当前只是把题目发给检索器。  
检索层没有强制 `source_file == 某文献.pdf`。

这在 2026 的 RAG 视角下是明显结构缺陷。  
指定 source 是 retrieval constraint，不应该交给 LLM 猜。

### 差距 4：layout 信息丢失

当前 chunk 只有文本，最多夹带图片路径。  
但 PDF 的页码、栏目、标题、图注、跨页关系没有结构化保存。

这会直接影响：

- 文献定位。
- 引用可信度。
- 图像生成时的画作/构图依据。
- 多页论述问题。
- 评测可解释性。

### 差距 5：LLM 生成内容与原文混用

当前：

```text
【全局上下文】...
【原文资料】...
```

一起存进 `contextual_chunk` 并返回给 Researcher。

问题是：`全局上下文` 是 LLM 生成的，不等同于原文。  
它可以增强检索，但不能被当成文献原文引用。

### 差距 6：入库失败默认放行

`extract_super_chunk()` 在 JSON 解析失败时默认：

```python
"is_domain_relevant": True
```

这会把解析失败 chunk 放进主索引。  
在 2026 的 RAG 工程里，这类样本应该进 quarantine，而不是默认成为知识库。

### 差距 7：ColBERT pickle 缺少版本治理

当前：

- Milvus DB 约 32 MB。
- `colbert_tensors.pkl` 约 6.3 GB。

这个缓存太重要，不能没有 manifest。  
它至少应记录：

- chunk_count
- chunk_id list checksum
- model path/revision
- parser version
- source PDF checksum
- build time
- embedding parameters

否则重建后很难确定 Milvus 和 ColBERT 是否一致。

## 更新后的重构目标架构

### 1. Evidence-first ingestion

入库前先生成 canonical evidence。

```text
PDF
  -> document manifest
  -> page records
  -> layout elements
  -> raw paragraphs
  -> stable chunks
  -> image records
  -> retrieval records
  -> Milvus / BM25 / Graph indexes
```

核心原则：

- 原文证据先落盘。
- LLM 增强后落盘。
- 向量索引可重建。
- 每个索引都有 manifest。

### 2. 分层字段设计

建议每个 chunk 至少有：

```json
{
  "chunk_id": "sha256...",
  "doc_id": "sha256...",
  "source_file": "明代山水画中桥梁意象研究.pdf",
  "title": "明代山水画中桥梁意象研究",
  "page_start": 12,
  "page_end": 13,
  "section_title": "桥梁意象的空间功能",
  "raw_chunk_text": "...",
  "contextual_prefix": "...",
  "retrieval_text": "...",
  "metadata": {
    "dynasty": ["明"],
    "painter": ["唐寅"],
    "concepts": ["桥梁", "流水", "游观路径"]
  },
  "quality": {
    "parse_status": "ok",
    "parser": "pymupdf+vlm",
    "char_count": 842,
    "garbled_ratio": 0.01
  }
}
```

区别：

- `raw_chunk_text`：最终可引用的证据。
- `contextual_prefix`：上下文增强，可辅助理解。
- `retrieval_text`：专门用于向量/BM25 检索，可包含 multi-query、HyDE。

### 3. Source-aware retrieval

检索前先解析 query：

```json
{
  "question": "根据《明代山水画中桥梁意象研究》...",
  "constraints": {
    "source_title": "明代山水画中桥梁意象研究",
    "source_file": "明代山水画中桥梁意象研究.pdf"
  },
  "entities": ["明代", "桥梁意象"],
  "task_type": "factual_qa"
}
```

然后：

- 如果指定文献存在，Milvus/BM25 检索强制 filter。
- 如果指定文献不存在，返回“资料库无该文献”。
- 如果没有指定文献，正常全库检索。

### 4. Dynamic evidence retrieval

不要固定 `final_k=3`。  
2026 的趋势是根据问题动态取证。

建议：

- factual_qa：同一文献 2-4 个 chunk。
- comparison：每个来源至少 2 个 chunk。
- evidence_missing：先做 source/entity check，再少量证据支持纠错。
- artist_prompt：证据 chunk + 图像/构图 chunk。
- concept_explain：术语精确 BM25 + dense 扩展。

### 5. Citation verifier

回答后必须验证：

```json
{
  "citation_valid": true,
  "all_citations_from_evidence": true,
  "required_source_cited": true,
  "page_chunk_exists": true,
  "false_premise_handled": true
}
```

如果失败：

- citation 错：重答。
- source 错：重新检索。
- evidence_missing 没先纠错：重答。
- 无证据：明确回答证据不足。

## 是否需要推倒重来

不需要。

应该保留：

- Milvus
- BGE-M3 dense/sparse
- ColBERT 思路
- RRF 融合
- reranker
- Neo4j 作为辅助召回
- contextual retrieval 思路
- reviewed 评测集

应该重构：

- PDF parse artifact
- page/chunk provenance
- source-aware retrieval
- raw/generated context 分层
- BM25/full-text
- citation verifier
- index manifest
- ingestion quarantine

应该暂停：

- DPO
- PPO
- GRPO
- 大规模 SFT
- 继续盲目扩充评测集

## 为什么现在不应该优先训练

训练只能学习已有系统的行为。  
如果当前系统给模型的证据不稳定，训练会把不稳定行为固化。

当前最大问题是：

- 检索到的 chunk 不一定来自指定文献。
- 引用没有页码/chunk。
- LLM 生成上下文可能被当作原文证据。
- 解析失败 chunk 可能进入主索引。
- 图像和文本关系没有结构化。

这些问题不是 SFT/DPO 能解决的。  
SFT 可以教模型“引用格式”，但不能让不存在的 page metadata 变出来。  
DPO 可以偏好“更像好答案”的输出，但不能保证答案基于正确 chunk。

## 2026 版优先级计划

### P0：证据层与文档层

目标：所有 chunk 可追溯。

任务：

1. 建 `data/processed/documents`。
2. 保存 `documents/pages/elements/chunks/images/manifest`。
3. 为 chunk 生成稳定 ID。
4. 保存 page number、section title、bbox 或 page region。
5. Milvus 入库只引用 canonical chunk。

验收：

- 任意 Milvus 命中都能回查到 `source_file + page + chunk_id + raw_text`。

### P0：source-aware retrieval

目标：指定文献题必须命中指定文献。

任务：

1. 建 title alias 表。
2. Query parser 提取《文献名》。
3. 检索加 source filter。
4. 指定文献不存在时返回明确错误。

验收：

- reviewed 集中“根据《xxx》”的题，检索结果必须包含 `xxx.pdf`。

### P0：raw evidence 与 generated context 分层

目标：引用只基于原文。

任务：

1. 拆 `raw_chunk_text`、`contextual_prefix`、`retrieval_text`。
2. 检索用 `retrieval_text`。
3. 回答 evidence pack 主体用 `raw_chunk_text`。
4. Prompt 明确哪些是原文，哪些是系统生成背景。

验收：

- 回答不得只引用 contextual_prefix 或 HyDE。

### P1：BM25/full-text 与 hybrid fusion

目标：术语、题名、画家名稳定命中。

任务：

1. 使用 Milvus full-text/BM25，或独立 BM25。
2. 题名、画家、术语建立 exact boost。
3. dense/sparse/BM25/ColBERT 做 RRF 或 weighted fusion。
4. 用 reviewed 集分任务评估。

验收：

- factual_qa、concept_explain、comparison 的 source hit 提升。

### P1：citation verifier

目标：引用不可伪造。

任务：

1. 检查 citation 是否来自本轮 evidence。
2. 检查 page/chunk 是否存在。
3. 检查指定文献是否被引用。
4. 检查错误前提是否先纠错。

验收：

- 引用不存在 PDF 的问题接近 0。
- 错误前提题不再出现“先否定后继续编”的合格误判。

### P1：ingestion quarantine

目标：坏 chunk 不进主索引。

任务：

1. JSON 解析失败不默认放行。
2. 进入 quarantine。
3. 输出入库质量报告。
4. unknown metadata 比例进入报告。

验收：

- 每次入库能看到 PDF 级、page 级、chunk 级质量统计。

### P2：多模态 RAG

目标：把 1131 张图片变成真正可检索证据。

任务：

1. 建 image records。
2. 保存 page、bbox、caption、linked chunk。
3. 增加 image embedding。
4. Artist prompt 可以引用图像证据。

验收：

- 图像相关创作建议能追溯到具体图片/页码/文献。

## 对当前评测和训练策略的影响

当前已经做的 reviewed 评测集仍然有价值，但使用顺序应调整：

1. 先完成 P0 重构。
2. 用 reviewed 集跑 retrieval baseline。
3. 如果检索 source/chunk 明显提升，再跑 answer baseline。
4. 如果 answer 失败集中在格式、拒答、引用表达，再做 SFT。
5. 如果 SFT 后偏好仍不稳，再考虑 DPO。
6. PPO/GRPO 目前不是必要路线。

## 最终判断

2026 最新资料强化了上一版结论：

**你的系统不是 RAG 思路错了，而是证据工程还没到可以训练模型的程度。**

下一步最正确的工作不是继续做评测集，也不是马上微调 Qwen3，而是先做：

1. canonical document store
2. page/chunk provenance
3. source-aware retrieval
4. raw/generated context 分层
5. BM25/full-text
6. citation verifier

这六件事做完以后，训练才有意义。否则训练只是在用模型弥补数据链路的问题，长期会越来越难排错。
