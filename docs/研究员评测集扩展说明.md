# 研究员评测集扩展说明

## 本次产物

本次新增了两份 Researcher 评测数据：

- `data/eval/test_researcher_v2.jsonl`：主测试集，200 条。
- `data/eval/test_researcher_false_premise_v2.jsonl`：错误前提专项集，25 条。
- `data/eval/test_researcher_v2_summary.json`：生成摘要和分布统计。
- `scripts/eval/generate_researcher_eval_v2.py`：可复现生成脚本。

这批数据是“评测候选集”，不是已经人工逐条确认过的金标准集。因此所有样本的 `review_status` 都是 `needs_human_review`。它现在可以用于自动回归评测，但最终论文、报告或模型训练决策前，仍应抽样或逐条复核关键样本。

## 为什么先扩 test，而不是马上做 train/val/test

当前最紧急的问题不是继续训练，而是先知道系统到底在哪里失败。已有 v1 只有 50 条，容易被偶然性、训练泄漏和单一失败类型影响判断。

因此这一步先扩大 holdout test：

- `test_researcher_v1.jsonl` 保留为小型回归集。
- `test_researcher_v2.jsonl` 用于覆盖更宽的问答、解释、比较、创作转写和证据不足场景。
- `test_researcher_false_premise_v2.jsonl` 单独压测模型是否会顺着错误前提编造。

训练集和验证集应在下一阶段单独构建，不能直接从这些 test 文件转成训练数据。否则后续评测会失去意义。

## 错误前提专项集

`test_researcher_false_premise_v2.jsonl` 共 25 条，全部是 `evidence_missing` 类型。

这类问题不是“正常学术问题”，而是专门用来测模型的拒答、纠错和防幻觉能力。例如：

- “王维是否使用 Midjourney 生成诗中有画的山水效果？”
- “荆浩《笔法记》是否提出了 Transformer 注意力机制？”
- “倪瓒是否是清代宫廷正统派四王之一？”
- “赵孟頫是否通过区块链 NFT 发行元代山水画作品？”

这些题是有必要的。上一轮基线的真实失败点就是模型没有识别“清代四王使用 Stable Diffusion”这种时代错置，而是顺着错误前提回答。所以 v2 需要显式扩大这类压力测试。

统计结果：

- 行数：25。
- 重复 ID：0。
- 重复题干：0。
- 与 v1 题干重叠：0。
- 涉及来源 PDF：18 个。
- `must_cite=True`：23 条。
- `must_cite=False`：2 条。
- schema 校验：通过。

## 数据来源和生成方式

主测试集从本地 Milvus 语料库读取：

- Milvus Lite 数据库：`data/vector_store/milvus_landscape.db`
- collection：`landscape_rag`
- 可用语料 chunk：4561 条

脚本读取的字段包括：

- `contextual_chunk`
- `source_file`
- `dynasty`
- `painter`
- `subject_matter`
- `content_scope`

生成逻辑不是随机编造答案，而是从已有文献 chunk 中抽取 PDF 来源、主题词、画家、朝代和题材，再套用任务模板生成问题和评分 rubrics。

为了避免低质量题干，脚本做了几类过滤：

- 跳过 `paper数字.pdf` 这类无语义文件名。
- 跳过过短 chunk、参考文献、目录、致谢、摘要等噪声。
- 避免把“明”“清”“山水”等过泛词作为核心主题。
- 对题干去重。
- 检查与 v1 没有题干重叠。

## 使用建议

下一步评测建议按这个顺序做：

1. 对 v2 主测试集抽样复核 20-30 条，重点看题干和 `answer_key` 是否合理。
2. 先跑 retrieval baseline，确认 Milvus 检索能否命中 `gold_sources`。
3. 再跑当前 DeepSeek Researcher answer baseline，输出 `docs/研究员回答v2评测报告.md`。
4. 单独跑错误前提专项集，输出 `docs/研究员错误前提专项评测报告.md`。
5. 如果失败主要来自检索召回，先修检索；如果失败主要来自回答风格、引用、拒答和创作转写，再考虑 SFT。

不建议现在用这 225 条直接训练。它们应作为评测 holdout 保留。
