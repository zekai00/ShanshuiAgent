# 检索基线评测报告

## 评测目的

本次评测只评估 researcher 工作流中的检索阶段，不评估最终回答生成质量。

目标是回答：

1. 当前检索服务能否根据问题找回相关文献。
2. 检索片段是否包含题目期望的关键实体。
3. 后续训练 Qwen3 researcher 之前，问题主要在检索还是回答生成。

测试集：

```text
data/eval/test_researcher_v1.jsonl
```

```text
50 条
```

## 总体结果

```text
Items attempted: 50/50
Source hit rate: 45/50
Average entity hits: 3.50
```

更准确地看，50 条中有 1 条 `evidence_missing` 的 `gold_sources` 为空，即“清代四王如何使用 Stable Diffusion 生成山水画”这类时代错置题。这类题不应该按普通 source hit 计算。

排除这条无 gold source 样本后：

```text

source hit rate: 45/49 = 91.84%
```

结论：当前检索链路整体可用，已经足够进入 researcher 回答生成 baseline 阶段。暂时不应该先训练模型，下一步应先评估“检索上下文给对以后，回答模型是否会忠于文献”。

## 低实体命中样本

只有 1 条样本实体命中低于等于 1：

```text
test_researcher_v1_043
请说明清代四王如何使用 Stable Diffusion 生成山水画。
```

检索结果：

```text
镣铐下的舞蹈——清代初期_四王_山水画现象透视.pdf
```

rerank score 为负数，说明检索器实际上认为相关性较低，但由于系统有兜底 Top-1 机制，仍返回了最接近的“四王”文献。

判断：

这是可接受行为。最终回答模型应该利用“四王”上下文纠正时代错置：清代四王不可能使用 Stable Diffusion，可转而说明四王的仿古笔墨与山水画程式。

## 主要发现

### 1. 检索器整体表现好于原始指标

原始 `45/50` 容易被误读为 90%。但实际 4 个 source miss 中，多数是 gold source 写得太窄，而检索结果本身相关。

按人工初步判断，当前检索质量已经足够支持下一步 researcher 回答评测。

### 2. gold_sources 需要人工复核

当前测试集是第一版，`review_status` 仍为 `needs_human_review`。本次评测暴露出 gold source 不是穷尽列表。

建议对以下题目人工复核并扩展 gold source：

```text
test_researcher_v1_022
test_researcher_v1_025
test_researcher_v1_034
test_researcher_v1_041
```

### 3. source hit 不能作为唯一指标

对于 RAG 系统，尤其是同一主题有多篇相关论文时，只看文件名是否命中 gold source 会低估实际检索质量。

后续评测应增加：

- `acceptable_sources`
- `context_support_score`
- `answer_supported_by_context`
- `citation_validity`

### 4. 下一步不应直接训练

目前检索已经能提供相对可靠的上下文。下一步应该评估当前回答模型是否会：

- 忠于检索上下文。
- 正确引用来源。
- 避免把检索资料外的信息写成事实。
- 在错误前提题中主动纠错。
- 在时代错置题中拒绝胡说。

这一步完成后，才能决定 Qwen3 researcher 的 SFT 数据应该重点覆盖哪些失败模式。

## 建议的下一步

### 第一步：修订评测集 gold source

人工复核 4 个 source miss 样本，决定是：

- 扩展 `gold_sources`；
- 增加 `acceptable_sources`；
- 或者改写题目，使 gold source 更明确。

### 第二步：做 researcher 回答基线

新增一个回答评测脚本：

```text
scripts/eval/run_researcher_answer_baseline.py
```

流程：

```text
question
  -> retrieve top 3
  -> 拼接检索上下文
  -> 调当前 researcher 回答模型
  -> 保存 answer、citations、used_sources、errors
```

输出：

```text
data/eval/baseline_researcher_answers_v1.jsonl
```

### 第三步：人工标注回答错误类型

建议错误标签：

```text
retrieval_miss
unsupported_claim
bad_citation
wrong_entity
wrong_dynasty
wrong_technique
missed_refusal
over_generalized
good
```

### 第四步：再做 SFT 数据

如果回答错误主要集中在“检索上下文正确但回答不忠实”，再构建 SFT 数据。

SFT 样本必须包含：

```text
用户问题 + 检索上下文 -> 标准回答
```

不要再做 `input` 为空的训练数据。

## 当前结论

当前检索链路通过了第一轮基线评测：

```text
50/50 成功运行
有 gold source 的样本 source hit = 45/49 = 91.84%
平均实体命中 = 3.50
事实问答、概念解释、artist_prompt 类任务表现稳定
主要问题是 gold source 标注不够完整，而不是检索器明显不可用
```

因此，下一步应进入 researcher 回答生成评测，而不是立刻进行 SFT、DPO 或 GRPO。
