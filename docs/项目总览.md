# ChineseLandscape 项目说明

## 项目定位

ChineseLandscape 是一个面向“中国传统山水画”的多模态 RAG 与多智能体创作系统。

它的目标不是做普通聊天，而是把山水画论文、论文插图、结构化元数据、向量检索、知识图谱、对话智能体和 ComfyUI 生图串成一条工作流：

1. 回答中国传统山水画相关的学术问题，并尽量给出文献依据。
2. 根据文献考据结果生成更符合山水画理论的 AI 绘画提示词和图像。

项目内部多处称它为“数字敦煌 V3.0”或“中国传统山水画多模态协作系统”。

## 现有数据

当前工作区已经包含一批可用数据和实验产物：

- `data/raw_pdfs/`：山水画相关 PDF 论文语料。
- `data/extracted_artworks/`：从 PDF 页面中裁剪出的插图或画作图片。
- `data/vector_store/`：Milvus Lite 向量库、ColBERT 张量和入库状态文件。
- `generated_images/`：ComfyUI/Flux 生成过的图像。
- `ComfyUI/`：内置的 ComfyUI 运行环境。
- `neo4j-community-2026.03.1/`：内置的 Neo4j 数据库。
- `data/fintune_results/`：领域模型回答实验结果，包括 base 与 LoRA 回答对比。

## 总体架构

项目主链路可以概括为：

```text
PDF 论文
  -> VLM 解析文字和插图
  -> LLM 生成全文大纲、语义切片和元数据
  -> BGE-M3 生成 dense/sparse/ColBERT 特征
  -> Milvus Lite 向量库
  -> Neo4j 知识图谱
  -> FastAPI 在线检索服务
  -> LangGraph 多智能体系统
  -> ComfyUI 图像生成
```

## 离线入库流程

入口文件是 `scripts/run_ingestion.py`。

它会扫描 `data/raw_pdfs/`，用 MD5 判断哪些 PDF 是新增或被修改的，然后执行增量入库或全量重建。

核心步骤：

1. `DocumentProcessor` 使用 VLM 逐页读取 PDF，提取正文和插图坐标。
2. 根据 VLM 返回的坐标裁剪图片，保存到 `data/extracted_artworks/`。
3. `LocalLLMExtractor` 调用长上下文模型为整篇论文生成全局大纲。
4. 本地 LLM 将正文切成更细的语义片段，并抽取朝代、画家、技法、题材、审美概念等元数据。
5. `BGEM3Engine` 为每个片段生成三类检索特征：dense 向量、sparse 词权重、ColBERT 多向量。
6. `LandscapeDatabaseManager` 将 dense/sparse 数据写入 Milvus Lite，并将 ColBERT 张量保存为 pickle。

相关模块：

- `src/ingestion/document_processor.py`
- `src/ingestion/llm_extractor.py`
- `src/ingestion/ingestion_manager.py`
- `src/retrieval/bge_m3_engine.py`

## 在线检索流程

入口文件是 `scripts/run_retrieval_server.py`。

它启动 FastAPI 服务，并暴露：

```text
POST /retrieve
```

检索核心在 `src/retrieval/online_retrieval.py`，由 `OnlineHybridRetriever` 负责。

检索不是单一路径，而是四路召回后融合：

- dense semantic search：语义相似度召回。
- sparse lexical search：关键词、术语和专名匹配。
- ColBERT search：token 级细粒度匹配。
- Neo4j graph search：根据朝代、画家、作品、技法等实体命中相关文本块。

四路结果会用 RRF 排位融合，再用 BGE-Reranker 精排，最后返回最相关的文献片段、来源 PDF、朝代、画家和分数。

## 知识图谱

知识图谱构建入口是 `scripts/build_full_graph.py`。

它从 Milvus 中读取所有 `contextual_chunk`，再用 `src/ingestion/linear_rag_graph.py` 抽取图谱结构。

图谱节点主要有三类：

- `Chunk`：文献切片。
- `Sentence`：切片内句子。
- `Entity`：实体，包括朝代、画家、流派、作品、技法、构图概念等。

图谱关系主要是：

```text
Chunk -[:HAS_SENTENCE]-> Sentence
Entity -[:MENTIONED_IN]-> Sentence
```

该图谱会参与在线检索，提升实体型问题的召回效果，例如“黄公望”“北宋”“披麻皴”“三远法”等。

## 多智能体系统

当前主要终端入口是 `scripts/run_chat_agent.py`。

智能体图定义在 `src/agent/graph.py`，使用 LangGraph 编排。

主要节点：

- `gateway`：入口节点，负责上下文门控。
- `summarizer`：长对话压缩，并沉淀用户偏好。
- `supervisor`：主管节点，只负责判断任务应该交给谁。
- `researcher`：学术研究员，调用资料库检索并输出文献化回答。
- `artist`：美术指导，把用户需求和研究员卷宗转成英文生图 prompt，并调用 ComfyUI。
- `chatter`：接待员，负责普通闲聊和非山水画问题。

核心工具在 `src/agent/tools.py`：

- `search_landscape_literature`：请求本地检索服务 `http://127.0.0.1:8000/retrieve`。
- `generate_landscape_image`：请求 ComfyUI `http://127.0.0.1:8188`，使用 `workflows/flux1_krea_dev-api.json` 生成图像，并保存到 `generated_images/`。

## 用户记忆

长期记忆模块在 `src/agent/memory/memory_manager.py`。

它使用 SQLite 保存每个 `user_id` 的结构化记忆：

- `preferences`：明确风格偏好，例如喜欢留白、南宋风格、浅绛设色。
- `feedback`：用户纠正和批评，例如不要太黑、不要太写实。
- `context`：最近项目上下文。

终端启动时会询问 User ID，然后加载该用户画像。

## Web UI

Web UI 文件是 `ui/web_ui.py`，基于 Gradio。

设计目标包括：

- 聊天交互。
- PDF 文献预览。
- Neo4j 图谱可视化。
- 底层日志显示。

但当前 Web UI 更像旧版或半迁移状态：它仍引用 legacy agent，并且存在导入顺序和路径问题。当前更可信的主入口是 `scripts/run_chat_agent.py`。

## DPO 与领域微调实验

项目中 `scripts/experiments/generate_dpo_data.py` 用来构造 DPO 偏好数据。

它的基本做法是：

1. 对一个山水画问题调用本地检索器拿到真实文献片段。
2. 用大模型基于检索文献生成一个较好的 `chosen` 回答。
3. 再生成一个故意敷衍、时空错乱或事实错误的 `rejected` 回答。
4. 将样本保存为 LLaMA-Factory 可用的 DPO 格式。

在本系统里，DPO 的目的不是替代 RAG，而是让领域模型更偏好以下回答方式：

- 基于资料库事实回答。
- 少编造画家、朝代、技法和作品关系。
- 更像专业中国美术史研究员。
- 回答中保留文献感、出处意识和术语解释。
- 避免网络化、敷衍化、泛泛而谈的回答。

因此，DPO 更适合作为 `researcher` 或最终回答模型的偏好对齐阶段，而不是用来训练检索器本身。

## 正常运行顺序

理论上的完整运行顺序：

1. 启动本地 VLM 服务，提供 `qwen-vl`，端口通常是 `8001/v1`。
2. 启动本地文本 LLM 服务，提供 `qwen3-4b-instruct`，端口通常是 `8000/v1`。
3. 运行 `python scripts/run_ingestion.py` 构建 Milvus/ColBERT 数据库。
4. 启动 Neo4j。
5. 运行 `python scripts/build_full_graph.py` 构建图谱。
6. 启动在线检索服务：`python scripts/run_retrieval_server.py`。
7. 启动 ComfyUI，监听 `8188`。
8. 启动终端智能体：`python scripts/run_chat_agent.py`。

需要注意：离线入库阶段和在线检索阶段都可能使用 `8000` 端口，但语义不同。入库阶段的 `8000/v1` 是本地 LLM；在线阶段的 `8000/retrieve` 是 FastAPI 检索服务。运行时要避免端口冲突。

## 当前工程状态

这是一个研究型原型系统，已经有比较完整的业务闭环，但还不是干净的生产级项目。

主要风险和待整理点：

- 缺少顶层 README。
- 多处硬编码绝对路径 `/root/Workspace/ChineseLandscape`。
- 配置和密钥管理需要清理，敏感值应全部迁移到 `.env` 并轮换。
- `ui/web_ui.py` 和部分 legacy 文件仍处于旧版或半迁移状态。
- `src/agent/subgraph_nodes.py` 引用了当前主线中不存在或已删除的旧对象，暂不应视为稳定入口。
- 仓库中存在未提交修改和删除文件，正式整理前需要确认哪些是用户有意改动。
- ComfyUI、Neo4j、大模型服务、检索服务之间依赖较多，启动顺序和端口规划需要文档化。

## 总结

ChineseLandscape 的核心价值在于：它把中国山水画的论文知识库、实体图谱、混合检索、多智能体问答和 AI 生图结合起来，使系统能够先查文献、做考据、理解画史和技法，再回答用户问题或生成更有传统山水画依据的图像。
